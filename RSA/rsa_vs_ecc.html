The Giants' Weakness
왜 RSA와 ECC로 영화를 못 보내는가?

"비대칭키 암호는 수학적으로 완벽하지만, 현실적인 용량, 변환, 속도의 한계가 있습니다."

1. 용량의 한계 (The Container Problem)

암호화 알고리즘은 데이터를 담는 '그릇(Container)'과 같습니다. 하지만 비대칭키의 그릇은 매우 작아서 \(M < N\) 조건을 만족해야 합니다.

RSA-2048 Container Max ~245 Bytes
ECC-256 Container Max ~32 Bytes

2. 변환의 난관 (The Mapping Problem)

"RSA는 정해진 틀(상수)에 넣으면 바로 답이 나오지만, ECC는 조건을 만족할 때까지 입력을 계속 바꿔야 합니다."

4. 하이브리드 방식의 차이 (Transport vs Agreement)

"대칭키(AES)를 공유하는 방식에서 RSA와 ECC는 결정적인 구조적 차이가 있습니다.
현대 보안(TLS 1.3)에서는 보안성이 더 뛰어난 ECC 방식을 표준으로 채택하고 있습니다."

RSA: 키 전송 (Key Transport)

"내가 만들어서 보내줄게"
  1. Alice: 혼자서 AES 키(K)를 생성함.
  2. Alice: Bob의 공개키로 K를 잠금.
  3. Network: 암호화된 키(📦)가 전송됨.
  4. Bob: 자신의 개인키로 상자를 열어 K 획득.

ECC: 키 합의 (Key Agreement)

"재료를 섞어서 같이 만들자" (ECDH)
  1. Both: 서로 공개 재료(Public Key)를 교환함.
  2. Alice: 내 비밀 재료 + Bob의 공개 재료 섞음.
  3. Bob: 내 비밀 재료 + Alice의 공개 재료 섞음.
  4. Math: 수학적으로 똑같은 AES 키(K)가 생성됨!

✅ 장점: 완전 순방향 비밀성 (PFS)

키 자체가 전송되지 않고 계산으로 만들어집니다. 매번 새로운 재료를 쓰면, 나중에 키가 털려도 과거의 대화는 안전합니다.

Feature RSA (Key Transport) ECC (ECDH Agreement)
AES 키 생성 한 명이 만들어서 보냄 (Delivery) 둘이서 같이 만듦 (Derivation)
키 노출 위험 전송 중에 탈취 가능성 존재 (암호화는 되어있음) 키 자체가 네트워크를 안 탐 (안전)
과거 보안 (PFS) ❌ 불가능 (마스터키 털리면 과거도 털림) ✅ 가능 (Ephemeral Key 사용 시 완벽 보호)
